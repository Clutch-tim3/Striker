```python
def get_offense_preview(self, payload: dict):
    """Preview offensive strategy details without admin unlock."""
    strategy_id = payload.get('strategy_id')
    if not strategy_id:
        # If no specific ID, get most recent for attack type
        attack_type = payload.get('attack_type')
        if attack_type:
            rows = self.db.execute(
                'SELECT * FROM offensive_strategies WHERE attack_types = ? '
                '   OR attack_types LIKE ? OR attack_types LIKE ?'
                'ORDER BY created_at DESC LIMIT 1',
                (attack_type, f'{attack_type},%', f'%,{attack_type}')
            ).fetchall()
        else:
            rows = []
    else:
        rows = self.db.execute(
            'SELECT * FROM offensive_strategies WHERE id = ?',
            (strategy_id,)
        ).fetchall()
    
    if rows:
        s = dict(rows[0])
        ats = [t.strip() for t in s.get('attack_types', '').split(',') if t.strip()]
        emit('OFFENSE_PREVIEW_DATA', {
            'strategy': {
                'id': s['id'],
                'name': s['name'],
                'description': s['description'],
                'attack_types': ats,
                'created_at': s['created_at'],
                'locked': s['locked'],
            },
            'contexts': {at: OFFENSIVE_CONTEXT.get(at, '') for at in ats},
            'playbooks': {at: DEFENSIVE_PLAYBOOK.get(at, '') for at in ats},
        })
    else:
        emit('OFFENSE_PREVIEW_ERROR', {'message': 'Strategy not found'})
```

## CRITICAL: Archive Synchronization & Database Lock Fixes

The current architecture has critical issues causing archive to show empty while dashboard shows 100+ antibodies:

### Root Cause Analysis

1. **No real-time archive updates**: Dashboard and Archive both poll `GET_ARCHIVE` every 5 seconds
   - Under rapid antibody creation (100+ in seconds), polling causes stale reads
   - Database locks from concurrent commits cause queries to fail/timeout
   - Archive page shows "database is locked" error

2. **Inefficient commit batching**: Each antibody creates a separate DB commit
   - 100 rapid antibodies = 100 commits in sequence
   - `Database.commit()` uses shared lock across all threads
   - Commit queue backs up, queries time out waiting for lock

3. **No push mechanism**: Frontend relies only on polling
   - Archive page has no way to know new data is available
   - Must wait for next poll cycle (up to 5 seconds)

### Solution: Real-Time Archive Sync with Optimized Commits

#### CRITICAL FIX: Database Lock Contention

The root cause of "database is locked" errors is the `Database.commit()` method using a SHARED LOCK (`self._lock`) across ALL threads, creating severe contention under load.

**Problem in db.py:**
```python
def commit(self):
    max_retries = 5
    for attempt in range(max_retries):
        try:
            with self._lock:  # ← SHARED ACROSS ALL THREADS!
                self.conn.commit()
                return
        except sqlite3.OperationalError as e:
            if 'database is locked' in str(e) and attempt < max_retries - 1:
                time.sleep(0.1 * (2 ** attempt))  # Too slow: 100ms, 200ms, 400ms
                continue
            raise
```

When 100 antibodies are created rapidly:
- Each calls `self.db.commit()`  
- All compete for the SAME lock
- First commit gets lock, others wait
- Lock released → next gets it → 100 commits in sequence
- Queries (`SELECT`) also need the lock → time out waiting
- Result: "database is locked" errors

**Fix 1: Reduce lock scope while preserving thread safety (db.py):**

The `self._lock` IS necessary for thread safety - SQLite connections are not thread-safe by default. The problem is commit contention, not the lock itself.

**Solution approaches:**

**Option A (Recommended): Batch commits to reduce lock acquisitions**
See Fix 3 below - only call `commit()` every N inserts instead of every insert.

**Option B: Separate connections per thread (overkill)**
Each thread gets its own DB connection. Too complex for this use case.

**Key insight:** WAL mode is ALREADY enabled and allows **concurrent reads** during writes. The "database is locked" error comes from:
1. Too many threads trying to `commit()` simultaneously (all waiting for the lock)
2. `commit()` holding the lock for longer due to disk I/O

**Fix 2: Keep WAL mode - it's already working correctly:**
```python
# In _init_conn (already correct):
conn.execute('PRAGMA journal_mode=WAL')  # ← Allows concurrent reads
conn.execute('PRAGMA synchronous=NORMAL')  # ← Balanced durability
```

**Fix 3: Batch commits in AntibodyStore (python/archive/antibody.py):**
```python
class AntibodyStore:
    def __init__(self, db: Database):
        self.db = db
        self._pending_count = 0
        self._max_batch_size = 10  # Commit every 10 inserts
    
    def create(self, threat: dict, response_taken: list, insights: dict = None) -> dict:
        # ... (creation logic)
        
        self.db.execute(INSERT_SQL, antibody)
        self._pending_count += 1
        
        # Only commit on batch boundary or in non-batch mode
        should_commit = (self._pending_count >= self._max_batch_size)
        
        if should_commit:
            self.db.commit()
            self._pending_count = 0
            
            # Emit real-time update
            emit('ARCHIVE_UPDATED', {
                'count': self.count(),
                'latest': {
                    'id': antibody['id'],
                    'type': antibody['attack_type'],
                    'severity': antibody['severity']
                }
            })
        
        logger.info(f'Antibody created: {antibody_id} ({antibody["attack_type"]})')
        return antibody
    
    def flush(self):
        """Force commit any pending inserts."""
        if self._pending_count > 0:
            self.db.commit()
            self._pending_count = 0
```

**Why this works:**
- 100 rapid inserts → ~10 commits (instead of 100)
- Lock acquired 10× less frequently
- WAL mode handles concurrent reads during the fewer commits
- Reads don't block on writes (WAL feature)
- Writes don't block reads (WAL feature)

**Fix 4: Faster retries on lock contention (db.py):**
```python
def commit(self):
    max_retries = 5
    for attempt in range(max_retries):
        try:
            with self._lock:  # Still needed for thread safety
                self.conn.commit()
                return
        except sqlite3.OperationalError as e:
            if 'database is locked' in str(e) and attempt < max_retries - 1:
                # Faster recovery: 10ms, 20ms, 40ms, 80ms, 160ms
                time.sleep(0.01 * (2 ** attempt))
                continue
            raise
```

**Result:** With batching + WAL + faster retries:
- 100 inserts = ~10 commits
- Each commit takes ~10ms with WAL (vs 100ms+ without)
- Total time: ~100ms for 100 inserts
- Reads succeed immediately (WAL)
